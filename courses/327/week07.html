The scaling limit of a data warehouse is when it can no longer function correctly with the current resources. Scaling can be done up (vertically) by adding more resources to the current machine, or out (horizontally) by adding more nodes or clusters of nodes to the current system. Scaling up is needed when the computing power of the current machine has reached its scaling limit.

Vertical Scaling

Implementation of scaling up is easy for software since the logic doesn't need to be broken up and put back together. There is a lower cost for total hardware, and lower energy use. However, vertical scaling is very limited when it comes to big data: there is no partitioning of data, there is only a single node. There is only so much RAM, CPU power and storage space that can go into one machine. Once that is used up, a multi-node architecture model is needed.

When scaling up, besides the physical limits, the downside includes downtime when the installation of upgraded hardware and probably new software to take advantage of the new hardwares' benefits is needed, not to mention the dollar and time cost of the installation, in most cases the vendor is locked in with their proprietary equipment and/or software, and future upgrading is finite.

Horizontal scaling

Performance, redundancy and consistency are the main benefits of scaling out. Using cloud-based distributed systems can be less expensive, both in time and money, as most of the configuration and security is part of the deal. Downtime due to upgrades is almost non-existent due to the redundancy factor inherent in distributed data. Processing is more consistent because of the computing resources being shared between nodes.

While the software licencing, technician salaries and hardware costs may be prohibitive for small to medium businesses, for the megacorporation, federal government agencies or other huge enterprises, the cost is bearable, even unavoidable, there are other limitations to scaling out also:

  • Utility costs will be higher if servers are onsite, due to heating and cooling necessary for the multitude of servers.
  • There are more requirements for space, networking equipment and dedicated staff.
  • Planning, implementing, configuring and maintaining all this equipment and software is extremely complicated.
  • The configuration, security, manpower time and dollar costs, space requirements and heating/cooling issues can be mitigated by using cloud services, but the financial cost may be even more than onsite.
  • Although it rarely happens, cloud systems can go down, which causes frustrating downtime with no other option than to wait until it comes back online.

Hadoop: A Distributed System

Hadoop has four main components:

  • HDFS, Hadoop Distributed File System, which handles file management
  • YARN, "Yet Another Resource Negotiator",Â which handles cluster management and job scheduling
  • MapReduce, which distributes the data among the nodes by its keys, allows functions to be processed on these partitions, then regathers the analyses, all while allowing for redundancy to avoid lost data and downtime
  • Hadoop Common, also known as Core, which is a framework of utilities and libraries to assist in connecting all the system's layers

All of these components work together to provide almost infinite scaling capabilities for huge data warehouses or lakes.
