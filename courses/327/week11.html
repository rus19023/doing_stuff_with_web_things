How does XQuery interface with MarkLogic document-store database?

​In MarkLogic, since the document-store model is a tree-type structure, a collection of documents (trees) is called a forest, which is the equivalent of traditional database tables. A collection of forests is similar to a traditional database.

MarkLogic provides a web-app GUI tool called Query Console to manage queries. It can be accessed through localhost:8000/qconsole in a web browser, where queries can be built from XQuery, JavaScript, SPARQL and SQL languages. It supports two dialects of its querying language, basically the original version and a ramped up, extended version with many new features. ​It gives access to XSLT, and offers libraries to be used with XQuery​ to facilitate interfacing and applications that use a variety of scripting languages.

Why is XQuery better than other options with a MarkLogic document-store database?

  • XQuery is supported by all major databases, making it useful for the data ingestion process from many sources.
  • XQuery is case-sensitive, and strings can be enclosed in single or double quotes, so it can do more flexible and specific searches​.
  • MarkLogic has a "universal" index set that is optimized for speed and created when data is ingested, for querying XML elements and JSON properties. This allows XQuery and XPath to be utilized to their full potential.​ The index "sees" structure (schema) the same way it "sees" content. Example: An XPath expression "/a /b /c" can be searched basically the same way as the phrase "a b c", because XPath 2.0 and XQuery 1.0 share the same data model.​
  • XPath 2.0 and XQuery 1.0 share the same data model, facilitating the simplicity of connecting varied data sources.
  • MarkLogic supports the major app-building languages, facilitating the creation of applications in the current language, but also allowing flexibility to change to a new system in the future.
  • XQuery has similar syntax to other query languages, such as SQL, PL/SQL and CQL, but are not limited to any particular database.
  • MarkLogic extends XQuery with features such as function mapping, serialization options, transaction separator of semi-colon and many others.
  • MarkLogic has the ability to do an excellent job with searching for words in text due to its upon-load universal index.

How does MarkLogic optimize indexing to support XQuery?

MarkLogic's indexing can specify collections by their keys, and has string range indexes and word dictionaries to specify their sort order​. The universal index works with the XML elements and JSON properties in loaded documents.

Some additional indexes to speed up specific types of searches​ are Word Indexing with inverted index for fast element word searches​, Phrase Indexing with phrase searches index for fast phrase searches to include two-word phrases, Relationship Indexing with parent-child index lets you search against an XPath even when you don't know the path in advance​. The index sees structure (schema) the same way it "sees" content. Example: An XPath expression "/a /b /c" can be searched basically the same way as the phrase "a b c".​

Summary

For XML document stores, or even when ingesting from traditional databases, MarkLogic is a very good source of ETL processing, for varied reasons. It is cost-effective with its different pricing plans for its cloud-based data hub services. It uses XQuery as the querying language, which only has moderate technical debt cost when the developers are not already familiar with SQL or any of its varieties. It is efficient in processing time, has flexibility with databases and application languages, and has very strong indexing power.
