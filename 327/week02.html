Kimball and Inmon Methodologies

Inmon Methodology

Bill Inmon's ideas for data warehouses work on a "bottom-up" principle. His model starts with the enterprise's data model, specifically what is most important for the business to run. The structure of the data from varied sources will be integrated and normalized to avoid duplication of data, help identify and prevent data anomalies. Once the infrastructure is in place, the data will be collected, analyzed, and stored in data marts built for each business purpose needed, possibly denormalized to facilitate reporting for decision-making. The warehouse is the only source of data for the data marts.

Inmon's methodologies are best for large-scale data warehouses. These data warehouses take many months to design and build, so they are not suitable for a quickly done project. It will be an expensive, long-term investment, but worth it in the long run. It should be more secure, as there are no external or non-standardized data collections. There will be costs in the future upkeep, as the data will grow and more employees will be needed to support the data warehouse.

Kimball Methodology

Ralph Kimball had different thoughts about big data storage and analysis. He was concerned that if the data was normalized, it would be difficult to analyze. He begins with finding the purpose for each main business purpose, and builds the marts for each of these purposes. The data warehouse is a conglomerate of these data marts built around the central repository of raw data, which comes from varies sources. Each of these marts becomes a point in the "star" pattern of the data warehouse.

Kimball's methodologies are best for small-scale data warehouses. These data warehouses are designed to be done quickly, and are suitable for a quick project. It will be less expensive and completed much faster. There will be minimal costs for future upkeep.

OLTP-Online Transactional Processing

Online Transactional Processing (OLTP) is a term for the process of processing transactions in a database. Historically, database transactions have been considered those related to business, such as financial transactions at a bank, purchases at a store, removing shipped products from inventory, etc. A newer definition of transactions would include any process that changes the state of the database.
